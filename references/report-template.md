# 报告模板

生成报告时，严格按以下结构输出。每节的写作指令必须遵守。

报告的核心原则：**展示思考过程，不只是结论。** 用户看到的不只是"是什么"，还有"我是怎么得出这个结论的"。

```markdown
# [项目名] 项目调研

## 1. 是什么

**写作指令：** 用一个具体类比让非技术人员也能理解它是什么。然后给出项目定位、技术栈、当前状态。不要只堆砌术语，要让读者 30 秒内建立直觉。

**结构：**

### 一句话定位

"[项目名] 就像是 XX 领域的 YY"（用读者熟悉的事物做比喻）

### 项目定位

- 它是什么：一句话核心功能
- 它不是什么：明确排除的范围（防止误解）
- 一句话总结："如果只能用一句话向同事介绍，我会说：……"

### 技术栈

语言、核心依赖、运行环境

### 当前状态

| 维度 | 数据 |
|------|------|
| Star 数 | |
| 最新版本 | |
| 最近更新 | |
| 协议 | |
| 主要语言 | |

> 注意：不要堆砌 GitHub 数据。Star 数、版本、协议就够了。不需要 contributor 数量、commit 频率等细节。

---

## 2. 为什么用（痛点 + 竞品 + 劣势）

**写作指令：** 先独立讲清楚"没有它之前，人们怎么忍受这个痛点"，再引入竞品对比。竞品对比必须同时列出优势和劣势——只比优势是营销，比劣势才是调研。

**结构：**

### 痛点：没有它之前

用具体场景描述，不是抽象描述。想一个具体的人，在什么时刻感到困扰，然后去搜索这样的工具？

> **苏格拉底追问：** "这个问题真的需要一个新工具解决吗？现有方案真的不能忍吗？" —— 如果答案是"其实还行"，要如实说明。

### 优势（3-5 个）

每个优势附具体证据（数据、代码引用、用户评价），不要空泛的形容词。

### 竞品对比

| 维度 | [项目名] | 竞品 A | 竞品 B |
|------|---------|--------|--------|
| 核心定位 | | | |
| 技术栈 | | | |
| 生态/社区 | | | |
| 适用场景 | | | |
| **它比竞品差在哪** | | | |

最后这一行"它比竞品差在哪"是必填项。诚实列出劣势，帮助读者做出平衡判断。

**分叉点分析：** 它和最接近竞品的"分叉点"在哪？同一个问题，它选了什么不同的路？

### 劣势与风险

- **维护活跃度：** 最近更新情况
- **社区健康度：** issue 响应速度
- **破坏性变更：** major version 升级频率和迁移难度
- **设计哲学的代价：** 它选择了 XX，牺牲了 YY

---

## 3. 怎么用

**写作指令：** 以"5 分钟内能让同事看到效果"为目标组织内容。先给最小路径，再扩展到常见场景。踩坑记录必须放在最后，不要让劝退信息挡住上手路径。

**结构：**

### 安装方式

一行命令，复制即用。

### 最小可运行示例

代码 + 运行命令，复制即用。目标：5 分钟内看到效果。

### 常见场景示例

2-3 个典型用法，每个不超过 10 行代码。

### 配置要点

只列最常用的配置项，不要穷举。

### 踩坑记录

已知的坑和解决方案，按影响程度排序。

> **苏格拉底追问：** "你能在 5 分钟内给同事演示这个项目的效果吗？" —— 如果不能，说明上手路径还不够清晰。

---

## 4. 内部实现（设计思路）

**写作指令：** 不是罗列架构图，而是讲清楚"作者为什么这样设计"。聚焦关键设计决策和取舍，让读者理解设计背后的思考过程。

**结构：**

### 核心抽象

这个项目用什么核心概念组织世界？（例：ora 用"定时器 + 字符串拼接"抽象终端动画）

### 关键设计决策（2-3 个）

每个决策包含：

| 决策 | 考虑的替代方案 | 为什么这样选 | 牺牲了什么 |
|------|--------------|------------|-----------|
| | | | |

### 数据流

用一句话描述核心流程：输入 → 处理 → 输出

### 作者动机

从 README、issue、博客中找动机。是解决自己的真实问题？技术兴趣？跟风？这个项目有没有作者的个人烙印？

> **苏格拉底追问：** "这个设计决策牺牲了什么？" —— 好的设计都是有代价的，找到代价才能真正理解决策。

---

## 5. 类型特化

**写作指令：** 根据项目类型补充特定维度。不同类型的项目，读者关心的东西不一样。

| 类型 | 侧重 |
|------|------|
| 框架/库/SDK | 包体积、tree-shaking、TS 支持、API 设计、迁移成本 |
| 工具类 | 性能基准 vs 同类、管道集成、shell 示例 |
| 平台类 | 自托管 vs SaaS、定价、数据导出、供应商锁定 |
| 学习资料 | 知识覆盖度、前置要求、学习路径 |
| Skill 类 | 触发条件、工作流步骤、输出格式、依赖工具 |

只写与当前项目类型对应的维度，不要全部列出。

> **苏格拉底追问：** "你当前的技术栈和场景，真的能用上它的这些特性吗？" —— 特性再好，用不上就是噪音。

---

## 6. 常见问题 QA

**写作指令：** QA 分两大类，都要写。重点是第二类——自主生成的现实问题。

### 第一类：用户交互问题

从本次调研会话中收集的所有 Q&A。每个 Q&A 记录原始问题和回答。

### 第二类：自主生成的现实问题（核心）

**这部分不是从 GitHub issues 中抄的。** 是调研者站在用户角度，自己提出现实中人们真正会问的问题，然后自己回答。

**生成方法：**

1. **站在不同角色提问：** 想象你是以下角色，每个角色会问什么？
   - 正在评估选型的技术负责人
   - 要在生产环境使用的开发者
   - 团队里持怀疑态度的同事
   - 预算有限的独立开发者
   - 刚入门的新手

2. **针对真实场景提问：**
   - "我们团队用的是 X 技术栈，能兼容吗？"
   - "这个项目多久没更新了，会不会突然不维护了？"
   - "我看到 Star 数很高，但实际用起来真的好用吗？"
   - "它和我现在用的 Y 相比，迁移成本高吗？"
   - "生产环境出问题了，社区能及时响应吗？"

3. **针对痛点和顾虑提问：**
   - "它最大的坑是什么？"
   - "什么情况下会后悔选了它？"
   - "有没有人用了之后又换回去了？为什么？"

4. **回答要求：**
   - 每个回答必须基于调研中获得的证据
   - 找不到证据的部分明确标注"未确认"
   - 回答要直白，不要回避问题
   - 如果答案是"确实有这个问题"，就如实说

**至少生成 8-10 个这样的 Q&A。** 数量不够说明思考不够深入。

---

## 7. 分析过程透明化

**写作指令：** 把调研过程中的 10 个核心问题的思考过程展示给用户。这不是附录，是报告的重要组成部分——让用户看到结论是怎么得出的。

**结构：**

展示以下 10 个问题的分析过程。每个问题包含：
- **问题：** 核心问题
- **思考动作：** 为回答这个问题做了什么（读了什么、搜索了什么、对比了什么）
- **结论：** 最终回答
- **追问：** 苏格拉底式追问，挑战这个结论

### 问题 1：它是什么？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "如果只能用一句话向不懂技术的朋友解释，我会怎么说？"

### 问题 2：没有它之前，人们怎么忍受这个痛点？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "这个痛点真的痛到需要一个新工具吗？"

### 问题 3：怎么用？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "5 分钟内能让同事看到效果吗？"

### 问题 4：有哪些竞品？为什么选它？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "它和最接近竞品的分叉点在哪？"

### 问题 5：它比竞品差在哪？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "它拒绝做的事情，是不是你恰恰需要的？"

### 问题 6：一句话讲清楚最核心的优势和特点

- **思考动作：** [用 4 个发现动作：分叉点分析、拒绝做什么、用户实际夸什么、如果消失了最怀念什么]
- **结论：** "它选择了 XX 哲学，牺牲了 YY"
- **追问：** "这个优势在什么场景下会变成劣势？"

### 问题 7：什么时候不该用？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "它的设计哲学在什么情况下会变成劣势？"

### 问题 8：解决了什么问题？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "用户实际描述的使用场景和 README 的定位一致吗？"

### 问题 9：作者为什么要做这个项目？

- **思考动作：** [描述具体做了什么分析]
- **结论：** [回答]
- **追问：** "这个项目有没有作者的个人烙印？"

### 问题 10：所以到底值不值得用？

- **思考动作：** [综合以上 9 个问题的回答]
- **结论：** [有条件推荐]
- **追问：** "如果这个项目消失了，用户真的会怀念吗？"

---

## 8. 总结

**写作指令：** 不要用"XXX 解决了 XX 场景下的 XX 问题"这种万能公式。给出有条件的、可操作的判断。

**结构：**

### 一句话判断

用最直白的话说清楚值不值得用。

### 推荐度

1-5 分（1=不推荐，3=有条件推荐，5=强烈推荐）

### 推荐条件

什么情况下应该用它（2-3 条具体条件）

### 不推荐条件

什么情况下不应该用它（2-3 条具体条件）

### 一句话总结

"如果你需要 XX，选它；如果你需要 YY，选别的。"
```

## 多渠道调研指南

在数据采集阶段，不要只依赖 GitHub。从多个渠道获取信息，交叉验证。所有数据采集统一使用 firecrawl。

### 渠道优先级

| 渠道 | 工具 | 说明 |
|------|------|------|
| GitHub | firecrawl scrape | 仓库页、README、releases、issues |
| Hacker News | firecrawl search | 技术深度讨论、架构评价 |
| Stack Overflow | firecrawl search | 常见问题、解决方案 |
| npm/PyPI | firecrawl scrape | 下载量、版本历史 |
| 小红书/知乎/掘金 | firecrawl search | 真实用户反馈、评测（SPA 站点） |
| Reddit | firecrawl search | 英文社区讨论 |
| 竞品调研 | firecrawl search | alternatives、vs |

### 搜索命令模板

**GitHub：**
```bash
firecrawl scrape https://github.com/<owner/repo> -o .firecrawl/github-repo.md --only-main-content
firecrawl scrape https://github.com/<owner/repo>#readme -o .firecrawl/readme.md --only-main-content
firecrawl scrape https://github.com/<owner/repo>/releases -o .firecrawl/releases.md --only-main-content
firecrawl scrape https://github.com/<owner/repo>/issues -o .firecrawl/issues.md --only-main-content
```

**中文社区：**
```bash
firecrawl search 'site:xiaohongshu.com "项目名"' -o .firecrawl/xhs.json --json
firecrawl search 'site:zhihu.com "项目名" 评测' -o .firecrawl/zhihu-review.json --json
firecrawl search 'site:zhihu.com "项目名" 体验/踩坑' -o .firecrawl/zhihu-feedback.json --json
firecrawl search 'site:juejin.cn "项目名"' -o .firecrawl/juejin.json --json
```

**英文社区：**
```bash
firecrawl search '"项目名" site:news.ycombinator.com' -o .firecrawl/hackernews.json --json
firecrawl search '"项目名" site:stackoverflow.com' -o .firecrawl/stackoverflow.json --json
firecrawl search 'site:reddit.com "项目名" review' -o .firecrawl/reddit-review.json --json
```

**包管理器：**
```bash
firecrawl scrape https://www.npmjs.com/package/<包名> -o .firecrawl/npm.md --only-main-content
firecrawl scrape https://pypi.org/project/<包名> -o .firecrawl/pypi.md --only-main-content
```

**竞品调研：**
```bash
firecrawl search '"项目名" alternative vs' -o .firecrawl/alternatives.json --json
firecrawl search '"项目名" limitations problems' -o .firecrawl/problems.json --json
```

### 信息交叉验证

- GitHub 上 Star 数很高但小红书/HN 上负面评价多 → 可能是营销驱动而非质量驱动
- GitHub issues 里都是功能请求但 Stack Overflow 上都是 bug → 文档不够好
- 官方文档说"简单易用"但社区讨论说"配置复杂" → 以社区为准

### 信息质量分级

| 级别 | 来源 | 可信度 |
|------|------|--------|
| 高 | 源码、官方文档、npm/PyPI 数据 | 直接证据 |
| 中 | GitHub issues、Stack Overflow、Reddit 高赞回答 | 社区验证 |
| 低 | 知乎个人体验、博客文章、未验证的评论 | 需交叉验证 |
